# Key Decisions

Lightweight ADR-style notes on the choices that shape Kubikles. Each is inferred
from the code and existing docs; the intent is the *why*, not the *how*.

## 1. Wails (Go + native webview) instead of Electron

- **Context** — The obvious way to build a cross-platform desktop dashboard is
  Electron, but it bundles a full Chromium runtime, costing startup time, memory,
  and download size.
- **Decision** — Build on Wails: a Go backend paired with the OS-native webview,
  producing a single compact binary.
- **Consequence** — Much smaller footprint and a native feel, and the Go backend
  can also run headless. The cost is dependence on platform webview libraries
  (e.g. WebKit2GTK on Linux) and the absence of a uniform bundled browser.

## 2. One backend core, three run modes behind an emitter

- **Context** — Users want a desktop app; operators also want a browser-served or
  in-cluster dashboard, and AI/automation wants headless access.
- **Decision** — Keep a single `App` core and abstract all real-time delivery
  behind an `events.Emitter`, so the same code runs over Wails IPC, over
  HTTP/WebSocket, or with no UI at all.
- **Consequence** — Desktop, server, and headless modes share one implementation
  with no per-mode forks in domain code. The cost is that every real-time feature
  must go through the emitter seam rather than calling a transport directly.

## 3. Code-generated method dispatch instead of reflection

- **Context** — Wails can bind Go methods via runtime reflection, but reflection
  keeps every method reachable and defeats the linker's dead-code elimination,
  inflating the binary.
- **Decision** — Generate an explicit dispatch table (`dispatch_gen.go`) from the
  `App` methods at build time.
- **Consequence** — Dead-code elimination stays effective and the binary stays
  small. The cost is a generation step that must be re-run when `App` methods
  change.

## 4. Coalesce watch events; watch only what's on screen

- **Context** — Busy clusters emit a firehose of watch events; forwarding each one
  individually across the IPC/WebSocket boundary would swamp the UI.
- **Decision** — Batch and deduplicate events into short time windows (flushing
  deletes immediately), and reference-count watchers so a resource is only watched
  while something is subscribed.
- **Consequence** — Predictable, bounded update overhead and no work for offscreen
  resources. The cost is a small added latency on non-delete updates and the
  bookkeeping of subscription lifecycles.

## 5. Helm behind a build tag

- **Context** — Helm support pulls in a large dependency surface, but not every
  user needs it.
- **Decision** — Gate Helm behind a `helm` build tag with stub implementations for
  the lite build.
- **Consequence** — Users can choose a smaller binary without Helm, or the full
  build with it. The cost is maintaining a stub surface that mirrors the real one.

## 6. Server mode ships without built-in auth

- **Context** — A browser-served dashboard needs an access model. Building
  authentication into the app would duplicate mature, existing tools.
- **Decision** — Ship server mode with no authentication of its own and document
  that it must run behind localhost, a tunnel, or an authenticating reverse proxy.
- **Consequence** — Simple, composable deployment that defers auth to purpose-built
  layers. The cost is a sharp, documented footgun: exposing the port directly
  grants full cluster access. See [Overview](overview.md#honest-boundaries) and
  [Server Mode](server-mode.md).

## 7. Pluggable AI providers with an allowlisted tool surface

- **Context** — An in-app AI assistant is most useful when it can read the cluster,
  but that is also where it becomes dangerous, and vendor lock-in is a risk.
- **Decision** — Put providers behind a common interface (Anthropic API and Codex
  CLI registered by default) and give the assistant only registered tools plus a
  command-prefix allowlist.
- **Consequence** — The product isn't tied to one AI vendor and the assistant's
  blast radius is bounded by the allowlist and cluster RBAC. The cost is
  maintaining the provider abstraction and the tool registry.

## 8. Pre-compressed frontend assets

- **Context** — The embedded frontend must be served quickly in both desktop and
  server modes without shipping a heavy runtime compressor.
- **Decision** — Gzip the frontend bundle at build time; desktop decompresses via
  Wails middleware, server mode serves the `.gz` directly.
- **Consequence** — Fast asset delivery with minimal runtime cost. The cost is a
  post-build compression step in the pipeline.
