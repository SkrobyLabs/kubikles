# Issue Detector

## What it does

Scan a cluster against a library of rules and report findings — each with a
severity, a category, the affected resource, and a suggested fix. Built-in rules
cover recurring problems across networking, workloads, storage, security, config,
and deprecations. Beyond the built-ins, you can drop your own rules into a local
directory as YAML files and reload them without restarting the app. Findings can be
grouped and browsed in the UI, and scans report progress as they run.

## Why it exists

A lot of cluster trouble is well-known and pattern-shaped — a service with no
matching endpoints, a workload with no resource limits, an API version about to be
removed. Rather than making operators rediscover each of these by eye, the detector
encodes them once and checks them in bulk. Making custom rules declarative (YAML,
no code, no rebuild) means teams can capture their own conventions without touching
the app.

## Notable constraints

- The detector reports; it does not fix. Suggested fixes are guidance for a human,
  not automated remediation.
- Custom YAML rules support a fixed set of check types; expressiveness is bounded
  by those types rather than being arbitrary code.
- A scan reflects cluster state at the moment it runs and only covers resources
  your credentials can read.

For the custom-rule format and check types, see
[Custom Issue Detector Rules](../custom-issue-rules.md) — the authoritative
reference, which this document deliberately does not duplicate.

Code entry point: `pkg/issuedetector/`.
