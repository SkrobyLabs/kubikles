# Port & Ingress Forwarding

## What it does

Establish and manage forwards from your machine into the cluster: port-forwards to
pods and services, and ingress forwarding that lets you reach a cluster ingress
locally. Forwards can be saved as configuration and reconnected, so the tunnels you
depend on come back after a restart or context switch rather than being re-created
by hand each session. Startup behaviour is configurable — restore favourites, all,
or none.

## Why it exists

Reaching an in-cluster service from a laptop is a routine need, and doing it with
raw `kubectl port-forward` means re-typing the same commands and losing them on
disconnect. Making forwards persistent, named, and managed removes that friction.
Browser-friendly (HTTPS) handling means forwarded endpoints work smoothly for web
UIs served from inside the cluster, including when Kubikles itself runs in server
mode.

## Notable constraints

- Forwards are only as durable as the underlying connection; if the target pod is
  replaced, the forward re-establishes against the new target where possible.
- Saved forwards live in local user settings, not in the cluster.
- A local port already in use will prevent that forward from binding.

Code entry points: `portforward.go`, `ingressforward.go`, `app_portforward.go`,
`app_ingressfwd.go`, and `frontend/src/features/portforwards/`.
