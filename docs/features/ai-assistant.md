# AI Assistant

## What it does

Chat with an AI assistant that can actually look at your cluster. Instead of a
generic chatbot, the assistant is wired to a registry of cluster tools — read pod
logs, fetch YAML, list resources, inspect events, trace dependencies, check RBAC,
pull metrics — so it can investigate a question rather than guess at it. Responses
stream token by token, tool calls are shown inline as they happen, and token usage
and cost are surfaced per turn. The same tool surface is exposed over an MCP server,
so the assistant can also be driven headlessly.

Providers are pluggable: the assistant talks to an AI backend through a provider
interface, with an Anthropic API provider and a Codex CLI provider registered out
of the box, and a model picker that groups the available models by provider.

## Why it exists

Diagnosing Kubernetes problems is exactly the kind of read-heavy, multi-step
investigation an assistant is good at — *if* it can see the cluster. Giving it a
constrained, allowlisted set of read-oriented tools lets it do real work (correlate
logs with events, follow a dependency chain) while keeping it inside guardrails.
Building on a provider abstraction avoids locking the product to one vendor.

## Notable constraints

- The assistant can only call registered tools, and its shell tool only runs
  commands matching a configured prefix allowlist — its reach is bounded by that
  allowlist plus your cluster RBAC, not by the model's discretion.
- It needs a configured, available provider (e.g. an Anthropic API key, or the
  Codex CLI installed); with none available the panel reports the provider as
  unavailable.
- Answers are model-generated and should be verified before acting on them,
  especially for any change to the cluster.

Code entry points: `app_ai.go`, `pkg/ai/` (providers and session manager),
`pkg/tools/` (tool registry and allowlist), `pkg/mcp/` (MCP server), and
`frontend/src/components/layout/AIPanel.tsx`.
