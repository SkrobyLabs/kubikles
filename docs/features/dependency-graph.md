# Dependency & Topology Graphs

## What it does

Render how a resource relates to the objects around it as an interactive graph.
Starting from any resource, Kubikles resolves its neighbourhood — owner references,
the pods a controller manages, the volumes and PVC/PV chain a pod mounts, the
services and endpoints that route to it, the config and secrets it consumes — and
lays it out visually. Nodes can be expanded to pull in the next ring of related
objects on demand. A separate node-topology view maps pods onto the cluster nodes
that host them.

## Why it exists

Kubernetes relationships are implicit, spread across labels, selectors, owner
references, and volume mounts. Reconstructing "what does this actually touch?" by
reading YAML is slow and error-prone. The graph makes the relationships explicit so
you can trace an impact or a dependency at a glance — the kind of question that is
hard to answer from a flat resource list.

## Notable constraints

- The graph is computed from the current state of the API, with request-scoped
  caching so a single expansion doesn't refetch the same objects repeatedly; it is
  a snapshot, not a live subscription.
- Relationships that are only expressible at runtime (e.g. a connection made
  purely by IP with no Kubernetes object linking them) can't be inferred.

Code entry points: `pkg/k8s/dependencies.go` and the `deps_*.go` resolvers,
`frontend/src/components/shared/DependencyGraph.tsx`, and
`frontend/src/features/cluster/topology/`.
