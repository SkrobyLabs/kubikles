# Workload Access: Logs, Shell & Files

## What it does

Reach into running workloads three ways, all without leaving the app:

- **Logs** — stream pod logs in real time. Log viewing handles the common cases
  directly: a single container, all containers of a pod, or all pods behind a
  workload, combined into one live stream.
- **Terminal** — open a full interactive shell into a container, with resizing,
  backed by a real pseudo-terminal.
- **Files** — browse a pod's filesystem, download individual files to your
  machine, and upload files into the pod (with progress feedback).

## Why it exists

"Read the logs," "exec in and poke around," and "grab that file" are the most
common follow-ups to "this looks wrong," so they are first-class rather than a
shell-out to `kubectl`. Doing this in-process also lets logs be batched and
delivered efficiently, the same way resource watch events are, so a chatty workload
doesn't overwhelm the UI.

## Notable constraints

- Terminal sessions are platform-specific under the hood — a PTY on Unix/macOS and
  conpty on Windows — but present the same behaviour to the user.
- Exec, log streaming, and file transfer all require the corresponding pod/exec
  permissions in your cluster role; without them the operation cannot start. File
  transfer additionally needs a usable shell / the expected tooling in the target
  container.
- In server mode the same streams flow over WebSocket instead of the desktop IPC
  bridge; native file dialogs degrade to browser download/upload (see
  [Server Mode](../server-mode.md)).

Code entry points: `pkg/k8s/podlogs.go`, `pkg/k8s/multilog.go` (log streaming),
`pkg/terminal/` (session lifecycle), `app_filetransfer.go` (pod file browse /
transfer), and `frontend/src/components/shared/` (`LogViewer.tsx`, `Terminal.tsx`).
