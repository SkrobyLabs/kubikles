# Helm Management

## What it does

Browse and manage Helm releases alongside native Kubernetes resources: inspect what
is installed, work with chart repositories, and pull charts from OCI registries.
Helm becomes just another thing you can see and act on in the same window as your
pods and services, rather than a separate CLI workflow.

## Why it exists

Many clusters are managed through Helm, so a client that ignored releases would
leave a blind spot next to the raw resources they create. Integrating Helm keeps
the "what's running and why" story complete.

## Notable constraints

- Helm support is a **build-time option.** The full build includes it; a lite build
  compiles in stub implementations instead, trading the feature for a smaller
  binary. This is a deliberate size-versus-capability choice — see
  [Decisions](../decisions.md).
- Operations act through Helm's own mechanics and honour the same release state and
  repository configuration Helm itself uses.

Code entry points: `app_helm.go` / `app_helm_stub.go`, `pkg/helm/`, and
`frontend/src/features/helm/`.
