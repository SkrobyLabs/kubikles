# Metrics & Monitoring

## What it does

Show resource usage — CPU and memory — for nodes, pods, and the workloads and
namespaces above them, both as current numbers and as history over time. Kubikles
reads live usage from the cluster's metrics-server, and, if a Prometheus is present,
uses it for richer historical charts. Prometheus can be auto-detected in the cluster
or configured manually, and the connection is remembered per Kubernetes context.
History views roll usage up along the natural hierarchy: per pod, per controller,
per namespace, and per node.

## Why it exists

"Is this thing under pressure, and was it always like this?" is a question a flat
resource list can't answer — you need numbers and a trend. Pulling metrics-server
data inline gives an at-a-glance current picture with no extra setup, while opting
into an existing Prometheus unlocks history without Kubikles having to store any
time-series of its own. Detecting and remembering Prometheus per context means the
richer view "just works" on clusters that have it.

## Notable constraints

- Live usage requires metrics-server (or an equivalent metrics API) installed in
  the cluster; without it, current CPU/memory figures are unavailable.
- Historical charts depend on a reachable Prometheus; with none configured or
  detected, the app falls back to current-only metrics.
- Figures reflect what the underlying metrics source reports and inherit its
  scrape interval and retention.
- Node CPU/memory can come from either metrics-server or Prometheus, and the two
  are reconciled to line up rather than measure different things. metrics-server
  reports the kubelet/cAdvisor working set (usage minus inactive file cache). The
  Prometheus path reconstructs the same value from node-exporter meminfo as
  `node_memory_AnonPages_bytes + node_memory_Active_file_bytes` — not
  `MemTotal - MemAvailable`, which excludes active page cache and reads roughly
  half. CPU is close between the two sources; the notable gap was memory.
- A node with no node-exporter instance (e.g. a tainted system pool the
  node-exporter DaemonSet doesn't tolerate) has no node-level Prometheus series.
  Its usage falls back to the per-container working-set sum, which omits memory
  not charged to a pod container (OS, kubelet, system daemons) and so reads lower
  than metrics-server. Scheduling node-exporter onto that pool is the only fix.

Code entry points: `app_perfmetrics.go`, `app_prometheus.go`, `pkg/k8s/metrics.go`,
the `pkg/k8s/prometheus*.go` files, and `frontend/src/features/cluster/metrics/`.
