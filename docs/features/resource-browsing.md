# Real-time Resource Browsing

## What it does

Browse the resources of one or more clusters in a live, sortable, filterable
table. Switch between kubeconfig contexts and scope the view to selected
namespaces. As the cluster changes, the list updates on its own — new pods appear,
statuses change, deleted objects disappear — without a manual refresh. Any resource
can be inspected and edited as YAML, and common actions (delete, scale, restart,
cordon, evict, and so on) are available inline.

Coverage spans the resource families an operator reaches for daily: workloads
(pods, deployments, statefulsets, daemonsets, replicasets, jobs, cronjobs),
cluster objects (nodes, namespaces, events, priority classes, webhooks), config
(configmaps, secrets, HPAs, PDBs, quotas, leases, limit ranges), storage (PVs,
PVCs, storage classes, CSI), network (services, ingresses, network policies,
endpoints), RBAC, and arbitrary custom resources via their CRDs.

## Why it exists

This is the core of the product — the fast, always-current window onto a cluster
that everything else hangs off. The design priority is that it stays responsive on
large, churning clusters, which is why updates are delivered through batched watch
events and each resource type is only watched while something is actually looking
at it.

## Notable constraints

- The list reflects exactly what your credentials can see; resources you lack RBAC
  to list simply won't appear.
- Live updates depend on the Kubernetes watch API; a resource kind with no watch
  support falls back to periodic listing.
- Editing writes YAML back through the API and is subject to the same admission
  and validation rules as `kubectl apply`.

Code entry points: `pkg/k8s/` (backend operations and watching),
`frontend/src/features/` and `frontend/src/components/shared/ResourceList.tsx`
(the universal table).
