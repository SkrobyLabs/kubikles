# Overview

## The problem

Working with Kubernetes day to day means constantly answering small questions —
*is this pod healthy, what's in its logs, what does it depend on, why won't it
schedule, what changed?* — across clusters and namespaces. The command line makes
each answer a separate incantation; heavyweight GUI dashboards answer them but at
the cost of a bundled browser runtime, hundreds of megabytes of memory, and
sluggishness on large clusters.

Kubikles exists to make those questions fast to ask and fast to answer, from a
single desktop window, without the weight. It targets developers and operators
who live in Kubernetes and want a responsive, native-feeling client that stays out
of the way.

## What it is (and isn't)

**It is** a read-and-operate client for clusters you already have access to. It
uses your existing kubeconfig (or in-cluster service-account credentials) and
speaks to the Kubernetes API through `client-go` with the permissions you already
hold. It covers the common resource kinds, live watching, editing, logs, exec,
forwarding, dependency visualisation, rule-based issue detection, optional Helm
management, and an AI assistant that can read the cluster to help you reason about
it.

**It isn't** a cluster provisioner, a CI/CD system, a GitOps controller, or a
policy-enforcement engine. It does not install or manage the cluster itself, does
not store cluster state of its own beyond local UI preferences, and does not grant
any access you don't already have — it is a lens over the API, bounded by RBAC.
The issue detector reports problems; it does not remediate them automatically. The
AI assistant is given read-oriented tools and an allowlist, not free rein.

## Honest boundaries

Because Kubikles can run as a browser-served dashboard as well as a desktop app,
its access guarantees depend on how it is run.

**Enforced**

- **Cluster permissions** are whatever the underlying kubeconfig / service account
  allows — Kubikles never escalates beyond the API's own RBAC.
- **AI tool scope** is constrained by an allowlist: the assistant can only invoke
  registered tools, and its shell tool only runs commands matching configured
  prefixes.

**Not a hard boundary**

- **Server mode has no authentication or authorization of its own.** When run as a
  headless/browser server it exposes full cluster access (including exec,
  port-forward, and secrets) to anyone who can reach the port. This is deliberate —
  it is meant to sit behind localhost, a VPN/SSH tunnel, or an authenticating
  reverse proxy. See [Server Mode](server-mode.md) for the authoritative security
  guidance; this document deliberately does not duplicate it.
- **Desktop mode inherits local machine trust** — the app runs as your user and can
  read your kubeconfig and local files like any other desktop program.
